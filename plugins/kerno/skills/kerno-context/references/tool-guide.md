# Tool approvals

Status, explanation, and impact analysis are read-only. Indexing reads repository files and writes Kerno’s local database. Task analysis, capsule creation, outcomes, decisions, routing, and comparison write only local Kerno state. Applying manual invalidation requires a non-dry-run call and explicit user intent.

`kerno_discover_models` starts the local Codex App Server with a fixed command, returns a sanitized live catalog snapshot, and may consume authenticated account capacity. It does not execute repository text or switch the current parent task. If App Server is unavailable, Plugin Mode returns an explicit unavailable/manual fallback; it must not invent a catalog or model recommendation. `kerno_route_task` remains a recommendation until an explicit Orchestrator turn accepts the selection, and the model remains requested-unconfirmed unless runtime evidence establishes an effective model or reroute.

The sixteen P0 tools are: `kerno_index_repository`, `kerno_repository_status`, `kerno_analyze_task`, `kerno_build_context_capsule`, `kerno_expand_context`, `kerno_explain_context`, `kerno_impact_analysis`, `kerno_record_decision`, `kerno_record_outcome`, `kerno_invalidate_context`, `kerno_discover_models`, `kerno_route_task`, `kerno_compare_runs`, `kerno_get_settings`, `kerno_update_settings`, and `kerno_render_panel`.

`kerno_render_panel` is the only presentation tool. It returns validated panel state and advertises the component resource. Codex renders that resource only when the plugin has been wired to a registered developer-mode app; the current local plugin has no app ID and therefore exposes structured output rather than inline UI. `kerno_get_settings` is read-only. `kerno_update_settings` writes only owner-controlled local Kerno settings, cannot enable telemetry, and never changes Codex account, sandbox, or model state.
